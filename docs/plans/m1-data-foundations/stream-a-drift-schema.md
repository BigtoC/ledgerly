## 1. Scope & non-goals

### In scope (Stream A builds)
- `lib/data/database/tables/*.dart` — one file per MVP table: `currencies_table.dart`, `transactions_table.dart`, `categories_table.dart`, `accounts_table.dart`, `user_preferences_table.dart`.
- `lib/data/database/daos/*.dart` — one DAO per table: `currency_dao.dart`, `transaction_dao.dart`, `category_dao.dart`, `account_dao.dart`, `user_preferences_dao.dart`. Thin SQL wrappers only — no business rules, no domain mapping.
- `lib/data/database/app_database.dart` — `@DriftDatabase` declaring `schemaVersion = 1`, all MVP tables, all MVP DAOs, constructor takes a `QueryExecutor` so tests inject `NativeDatabase.memory()`.
- `drift_schemas/drift_schema_v1.json` — first schema snapshot, generated by `drift_dev schema dump`.
- Generated code (`app_database.g.dart`, per-table `*.g.dart` where Drift emits one) committed alongside.

### Out of scope (Stream A does NOT build)
- **Freezed domain models** in `lib/data/models/` → **Stream B** (Agent B). Drift data classes stay inside `lib/data/database/`; they never cross the repository boundary (`CLAUDE.md` → Architecture Non-Negotiables).
- **Repositories** (`lib/data/repositories/*`) → **M3**. Stream A does not map Drift rows to Freezed models and does not enforce business rules (category type-lock, archive-vs-delete, etc.).
- **`LocaleService`** (`lib/data/services/locale_service.dart`) → **Stream B**.
- **Seed data** (currencies, categories, Cash account) → **M3**, driven from repositories. No seed SQL in this stream.
- **Phase 2 tables** (`exchange_rates`, `pending_transactions`, `wallet_addresses`). Not declared, not imported, not referenced. Schema bumps to v2 in Phase 2.
- **`pubspec.yaml` edits** — Drift deps already present from M0.

## 2. Table definitions

Column names, types, constraints, and notes mirror `PRD.md` lines 259–414. This section covers only the **Drift translation** (Dart property names, Drift column builders, indexes, and the tricky bits). **Do not re-copy the PRD schema into code comments** — cite `PRD.md` line ranges in class-level dartdoc.

Drift conventions we adopt, repo-wide:
- Table classes are **plural** (`Currencies`, `Transactions`, `Categories`, `Accounts`, `UserPreferences`).
- Data classes are **singular** via `@DataClassName('Currency')`, `@DataClassName('TransactionRow')`, `@DataClassName('CategoryRow')`, `@DataClassName('AccountRow')`, `@DataClassName('UserPreferenceRow')`.
  - Rationale: `Transaction`, `Category`, `Account` are reserved for the **Freezed domain models** (Stream B). We prefix Drift data classes with `…Row` so a file that imports both does not collide. `Currency` is fine unprefixed because Stream B's Freezed model will also be called `Currency` and the two never appear in the same file (repos own the only crossing).
- Drift column names in SQL are the **snake_case** version of the Dart property (Drift's default). We do **not** override with `.named(...)` unless the PRD column spelling demands it — see the amount column below.
- Boolean defaults use `.withDefault(const Constant(false))`. Integer defaults use `.withDefault(const Constant(0))`. Timestamps use `DateTimeColumn` (Drift stores as Unix ms).

### 2.1 `currencies` — PRD 259–273

```dart
@DataClassName('Currency')
class Currencies extends Table {
  TextColumn get code => text()();                          // ISO 4217 / token symbol
  IntColumn get decimals => integer()();                    // SSOT for minor-unit scale
  TextColumn get symbol => text().nullable()();             // '$', '¥', 'NT$', ...
  TextColumn get nameL10nKey => text().nullable()();        // sql: name_l10n_key
  BoolColumn get isToken => boolean().withDefault(const Constant(false))();
  IntColumn get sortOrder => integer().nullable()();

  @override
  Set<Column> get primaryKey => {code};
}
```

Notes:
- `code` is the **natural primary key**. No autoincrement.
- `decimals` is the **single source of truth** for minor-unit scaling (`CLAUDE.md` → Data-Model Invariants). Never duplicated onto transaction rows.
- No `is_archived`; seeded fiat rows are permanent.
- MVP seeds USD/EUR/JPY/TWD/CNY/HKD/GBP at M3. No tokens in MVP.

### 2.2 `transactions` — PRD 275–291

```dart
@DataClassName('TransactionRow')
class Transactions extends Table {
  IntColumn get id => integer().autoIncrement()();
  // Integer minor units. Scaling factor is `currencies.decimals` — never a double.
  IntColumn get amountMinorUnits => integer().named('amount_minor_units')();
  TextColumn get currency =>
      text().references(Currencies, #code)();
  IntColumn get categoryId =>
      integer().named('category_id').references(Categories, #id)();
  IntColumn get accountId =>
      integer().named('account_id').references(Accounts, #id)();
  TextColumn get memo => text().nullable()();
  DateTimeColumn get date => dateTime()();
  DateTimeColumn get createdAt => dateTime().named('created_at').nullable()();
  DateTimeColumn get updatedAt => dateTime().named('updated_at').nullable()();
}
```

Notes:
- `amountMinorUnits` uses `.named('amount_minor_units')` because PRD pins the **SQL column name** and migration tests compare SQL snapshots. The Dart property stays camelCase.
- `.references(...)` on `currency`, `categoryId`, `accountId` emits SQL `REFERENCES` constraints. Foreign keys fire at the SQLite engine level when `foreign_keys = ON` is set on the DB connection (done in `AppDatabase` — see §4).
- Transaction **type** (expense/income) is **not a column** — it is derived from the linked category's `type`. Do not add a `type` column.
- **Indexes** (Drift's `@TableIndex` on the table class):
  - `transactions_date_idx` on `date DESC` — Home query sorts by day. Required because the Home stream pages up to 10 000 rows.
  - `transactions_account_idx` on `account_id` — Accounts tab balances + archive checks.
  - `transactions_category_idx` on `category_id` — category type-lock check + archive checks.
  ```dart
  @DataClassName('TransactionRow')
  @TableIndex(name: 'transactions_date_idx', columns: {#date})
  @TableIndex(name: 'transactions_account_idx', columns: {#accountId})
  @TableIndex(name: 'transactions_category_idx', columns: {#categoryId})
  class Transactions extends Table { ... }
  ```

### 2.3 `categories` — PRD 293–314

```dart
@DataClassName('CategoryRow')
class Categories extends Table {
  IntColumn get id => integer().autoIncrement()();
  TextColumn get l10nKey =>
      text().named('l10n_key').nullable().unique()();         // stable identity for seeded rows
  TextColumn get customName => text().named('custom_name').nullable()();
  TextColumn get icon => text()();                            // string key → icon_registry
  IntColumn get color => integer()();                         // index into color_palette
  TextColumn get type => text()();                            // 'expense' | 'income'
  IntColumn get parentId =>
      integer().named('parent_id').nullable()
        .references(Categories, #id)();                       // self-FK, nullable root
  IntColumn get sortOrder => integer().nullable()();
  BoolColumn get isArchived =>
      boolean().named('is_archived').withDefault(const Constant(false))();
}
```

Notes:
- `l10nKey` is `UNIQUE` but **nullable** — custom categories created by the user have `l10n_key = NULL` and `custom_name = 'My thing'`. SQLite treats multiple `NULL`s as distinct under `UNIQUE`, which is exactly what we want.
- `parent_id` is the **self-FK** for subcategories (Food → Groceries). Nullable because top-level categories have no parent. Self-FKs in Drift need the owning table name on both sides — `.references(Categories, #id)` works because `Categories` is the class we're defining.
- **`type` is stored as TEXT**, not an enum. Drift enum mapping lives in Stream B's Freezed model; Stream A only stores/retrieves the raw string. This keeps the DAO layer independent of the enum's package location.
- **`type` immutability is NOT enforced at the table level** — SQLite cannot express "frozen after first referencing row". Enforcement lives in `CategoryRepository` (M3). Stream A simply stores whatever it is handed.
- No `CHECK(type IN ('expense','income'))` constraint — kept out to avoid migration pain if we add a third type in Phase 2 (e.g., transfer). Validation lives in the repository.
- **Indexes**: `@TableIndex(name: 'categories_parent_idx', columns: {#parentId})` for subcategory lookup.

### 2.4 `accounts` — PRD 315–334

```dart
@DataClassName('AccountRow')
class Accounts extends Table {
  IntColumn get id => integer().autoIncrement()();
  TextColumn get name => text()();
  TextColumn get type => text()();                            // 'cash' | 'bank' | 'other'
  TextColumn get currency =>
      text().references(Currencies, #code)();
  IntColumn get openingBalanceMinorUnits =>
      integer().named('opening_balance_minor_units').withDefault(const Constant(0))();
  TextColumn get icon => text().nullable()();                 // string key → icon_registry
  IntColumn get color => integer().nullable()();              // palette index, nullable
  IntColumn get sortOrder => integer().nullable()();
  BoolColumn get isArchived =>
      boolean().named('is_archived').withDefault(const Constant(false))();
}
```

Notes:
- `openingBalanceMinorUnits` is `INTEGER` with SQL column name pinned. Same reasoning as `transactions.amount_minor_units`.
- Tracked balance is **derived**, not stored. No `current_balance` column.
- **No CHECK constraint** on `type` — enforcement lives in `AccountRepository`.

### 2.5 `user_preferences` — PRD 401–414

```dart
@DataClassName('UserPreferenceRow')
```

## 3. DAO contracts

One file per table in `lib/data/database/daos/`. All DAOs are thin SQL wrappers: they accept Drift rows / `Insertable`s, they return Drift rows, and they do nothing else. **Freezed mapping and business rules land in M3 repositories — never here.**

Common shape:

```dart
@DriftAccessor(tables: [Transactions])
class TransactionDao extends DatabaseAccessor<AppDatabase> with _$TransactionDaoMixin {
  TransactionDao(super.db);
  // methods below
}
```

### 3.6 Return-type rule (non-negotiable)

**Every DAO method returns a Drift data class** (`Currency`, `TransactionRow`, `CategoryRow`, `AccountRow`, `UserPreferenceRow`) or an `Insertable`/`Companion`/`int` primitive. **Never** a Freezed domain model. The Drift → Freezed translation happens **inside M3 repositories** (`CLAUDE.md` → Architecture Non-Negotiables). `import_lint` rules from M0 block DAO imports outside `data/repositories/`.

## 4. `AppDatabase`

**Location:** `lib/data/database/app_database.dart`.

```dart
import 'package:drift/drift.dart';
// part directive emitted by build_runner:
part 'app_database.g.dart';

@DriftDatabase(
  tables: [
    Currencies,
    Transactions,
    Categories,
    Accounts,
    UserPreferences,
  ],
  daos: [
    CurrencyDao,
    TransactionDao,
    CategoryDao,
    AccountDao,
    UserPreferencesDao,
  ],
)
class AppDatabase extends _$AppDatabase {
  AppDatabase(super.executor);

  @override
  int get schemaVersion => 1;

  @override
  MigrationStrategy get migration => MigrationStrategy(
        onCreate: (m) => m.createAll(),
        onUpgrade: (m, from, to) async {
          // MVP ships at v1; no upgrade steps exist.
          // Phase 2 adds v1→v2 here (pending_transactions, wallet_addresses,
          // exchange_rates, + token rows). Do NOT add v2 shapes in MVP.
        },
        beforeOpen: (details) async {
          await customStatement('PRAGMA foreign_keys = ON');
        },
      );
}
```

Key points:
- **Constructor takes a `QueryExecutor`** (not a file path). Tests pass `NativeDatabase.memory()`; production passes `drift_flutter`'s `driftDatabase(name: 'ledgerly')` builder (assembled in `app/bootstrap.dart` at M4, injected as a `Provider` override). Stream A does not wire the production executor — that is M4's job.
- **`foreign_keys = ON`** in `beforeOpen` so `.references(...)` constraints actually fire. SQLite defaults them off per connection. Without this, the M3 repository tests for FK integrity silently pass.
- **No seeding** in `onCreate`. `m.createAll()` creates tables and indexes — that is all. Seeding is a repository-level operation at M4 bootstrap (M3 deliverable).
- Generated superclass `_$AppDatabase` comes from `build_runner`. DAO mixins (`_$TransactionDaoMixin`, etc.) are also generated; the DAO files above reference them.

## 6. Code-gen workflow

Every change to a table class, `@DataClassName`, `@DriftDatabase`, or `@DriftAccessor` requires:

```bash

```

Rules:
- Generated `*.g.dart` files are **committed** (matches repo's existing M0 posture).
- Use `dart run build_runner watch --delete-conflicting-outputs` during active development.
- If `flutter analyze` reports missing `_$AppDatabase` or `_$FooDaoMixin`, codegen is stale — run build_runner, do not write the symbols by hand.
- After the v1 snapshot is dumped, re-run codegen if any annotation changed, then re-dump the snapshot. The snapshot must reflect exactly what `AppDatabase` compiles to.

## 7. Money-storage enforcement

Every monetary column is `IntColumn()`. In Stream A that is exactly two columns:

| Column                                   | Table          | Type         | SQL name                         |
|------------------------------------------|----------------|--------------|----------------------------------|
| `amountMinorUnits`                       | `transactions` | `IntColumn`  | `amount_minor_units`             |
| `openingBalanceMinorUnits`               | `accounts`     | `IntColumn`  | `opening_balance_minor_units`    |

Both carry a **dartdoc comment** on the getter referencing `currencies.decimals` as the scaling factor, e.g.:

```dart
/// Integer minor units. Scaling factor is `currencies.decimals` — never a
/// double. See `PRD.md` → Money Storage Policy and `CLAUDE.md` →
/// Data-Model Invariants.
IntColumn get amountMinorUnits => integer().named('amount_minor_units')();
```

No `RealColumn`, no `real()`, no `double`, ever — not in tables, not in DAOs, not in generated code. The M1 exit-criteria grep (§8) is the safety net.

## 8. Exit criteria (from M1 plan)

Stream A is done when **all** of the following hold:

1. `flutter analyze` — clean (no errors, no lints).
2. `flutter test` — passes. There are no behavioural tests in M1 (per `implementation-plan.md` §7); this verifies the tree compiles with generated Drift code.

4. `dart run build_runner build --delete-conflicting-outputs` — idempotent: running twice produces no diff.
5. **Money grep:** from repo root, `grep -rnE 'double\s+\w*(amount|balance|rate|price)' lib/` returns **zero hits** (per guardrail G4). Stream A's only exposure is the two money columns above; grep must not flag anything in `lib/data/database/`.
6. `AppDatabase` constructor accepts a `QueryExecutor` (verifiable by line inspection). No production `driftDatabase(...)` call lives inside `app_database.dart`.
7. All five DAOs exist, all five tables are registered in `@DriftDatabase`, all four indexes (§2.6) are declared.

### 9.2 Constraints Stream A imposes on Stream B

- **`amountMinorUnits` and `openingBalanceMinorUnits` are `int` everywhere** — Freezed model, domain model, controller, widget. No `double`, not even for display (display formatting is the M2 `money_formatter`'s job).
- **`currency` is a `String` code at the Drift layer.** Stream B must decide: does the Freezed `Transaction` model carry a `String currencyCode` or a nested `Currency` object? Repositories (M3) will join if the latter. Stream A has no preference, but we need a contract (see §10).
- **`type` on `CategoryRow` and `AccountRow` is `String` at the Drift layer.** Stream B is expected to wrap with an enum (e.g. `CategoryType.expense / income`). The **string values** stored in SQLite are `'expense'` / `'income'` (categories) and `'cash'` / `'bank'` / `'other'` (accounts). Changing the wire values later is a migration, not a refactor.
- **Drift data-class names are `…Row` (except `Currency`).** Stream B's Freezed classes should **not** be suffixed — the whole point of `@DataClassName('TransactionRow')` is to keep the clean name `Transaction` reserved for the domain model.
- **Serialization is Dart-side only.** Drift emits no `toJson` for the DB; user-facing JSON (backup/restore in M6, if ever) is Stream B's Freezed territory. Stream A does not pick a `@JsonKey` snake_case policy.

### 9.3 Naming choice that constrains Stream B

If Stream B reaches for `json_serializable` with default `fieldRename` and expects `amount_minor_units` in JSON while Drift's Dart property is `amountMinorUnits`, the two disagree **only at the JSON boundary** — which does not exist in MVP. So: **no current constraint, but flag if/when backup/restore lands.**

## 10. Open questions

Decisions needed from a human before Stream A merges (or in tandem with Stream B's first PR):

1. **Transaction currency modelling in Freezed (Stream B).** Does `Transaction` carry a raw `String currencyCode`, or a nested `Currency` object resolved by the repository? Stream A stores the code; either works. A nested object costs one extra join in `TransactionRepository.watchAll()`; a string defers formatting knowledge to the controller. **Recommendation:** nested `Currency` on read (join in the repo), string on write (the form only knows the code the user picked). Needs sign-off from Agent B.
2. **Account `type` values.** PRD lists `'cash' | 'bank' | 'other'` (line 321). Confirm the exact string literals before Stream B writes the enum mapping — once seeded rows exist, these values are migration-locked.
3. **`CategoryRow.type` check constraint.** PRD says `NOT NULL — 'expense' or 'income'` but the proposed table omits a SQL `CHECK` (§2.3). Confirm with platform lead that repository-level enforcement is acceptable (Stream A's default assumption). Adding a `CHECK` later is a schema bump.
4. **Drift `created_at` / `updated_at` population.** PRD marks them nullable with no default. Options: (a) Drift `.clientDefault(() => DateTime.now())`, (b) leave nullable and populate in the repository. **Recommendation:** populate in the repository — `clientDefault` does not fire for `UPDATE`, so `updatedAt` would go stale. Needs sign-off.
5. **Index on `categories(l10nKey)`.** We rely on the existing `UNIQUE` constraint's implicit index. Confirm no additional index is needed for seed idempotency (read volume is once per launch on an empty DB).
6. **`UserPreferenceRow.value` format.** PRD says "JSON-encoded" (line 406). Confirm the convention is "always JSON, even for scalar bools/strings" so the repository can single-code-path its parsing. Affects only M3, but worth documenting alongside the table definition comment.
